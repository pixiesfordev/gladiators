Easy grid setup and scripting for Unity 3D
Getting Started

Setting up a new grid

Choose a GameObject in your scene that will carry the grid. Go to Component → Grid Framework → Grids and choose the grid component you want to add. Alternatively you can create a grid from scratch by going to GameObject → 3D Object → Grid. Regardless of which method you choose, a fitting renderer will be added to the GameObject as well. Once you have your grid in place you can then position it by moving and rotating the GameObject, you can change the settings and you can reference it through scripting.

Creating grids programmatically

All grids inherit from the GridFramework.Grids.BaseGrid class, which in turn inherits from Unity's own MonoBehaviour class (the class of all scripts). As such grids can be created programmatically the same way as any other component in Unity.

The usual approach is to use the AddComponent method of the GameObject class to add a new grid to the scene

GameObject go // object to carry the grid
RectGrid grid = go.AddComponent<RectGrid>(); // reference to the new grid

Here we use the rectangular grid as an example, but any grid works the same way. Also keep in mind that the abstract grid classes cannot be instantiated, they only serve as abstraction.

Referencing a grid

You can reference a grid the same way you would reference any other component:

var myGrid = myGameObject.GetComponent<BaseGrid>(); // C#
var myGrid = myGameObject.GetComponent.<BaseGrid>(); // UnityScript

This syntax is the generic version of GetComponent which returns a value of type Grid instead of Component. See the Unity Scripting reference for more information. You can access any grid variable or method directly like this:

var myVector3 = myGrid.WorldToGrid(transform.position);

If you need a special variable that is exclusive to a certain grid type (like Spacing for rectangular grids) you will need to use that specific type rather than the more general BaseGrid:

var myGrid = myGameObject.GetComponent<RectGrid>();

Drawing the grid

A grid can be drawn using gizmos to give you a visual representation of what is otherwise infinite in size. Note that gizmos draw the grid in the editor, but they do not render the grid in a finished and published game during runtime. If you wish the grid to be visible during runtime you need to render it.

In the editor you can either hide the grid altogether by disabling its renderer or hide any individual axis by setting the alpha value of its colour to zero.

Legal number values

Some numbers, like the Spacing for rectangular grids, the Radius of polar- and hex grids or the Depth of layered grids are restricted in what values are legal. The documentations lists the restrictions, but keep in mind that even if an extreme value is legal that does not guarantee that the result will also be legal.

Case in point: the minimum legal value for the Smoothness of a polar grid is 1, while the maximum is unbounded. Try it out: create a polar grid in scene view, make sure it is set to relative size and then drag the Smoothness slider all the way to the right. Your grid will get denser and denser until you eventually get an Overflow Exception. What has happened? Because your grid got more and more dense the number of segment lines to draw eventually got larger than the size of an integer on your hardware, which causes an exception in C#.

A number is only legal in the sense that it is mathematically a legal number. You are still responsible for the result of your computations. As we have seen, whether an extreme value will cause problems or not depends on the context in which the value is used, so it would make no sense to make it illegal always just because it can break in some context. In reality though you will never really need such extreme numbers, I am only writing this for the sake of completeness.
